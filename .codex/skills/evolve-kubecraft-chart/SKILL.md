# Evolve Kubecraft Chart

Extend the chart without turning `values.yaml` into a direct copy of Kubernetes manifests. Keep common configurations concise, infer relationships that are unambiguous, and expose native API fields for advanced cases.

Read [references/project-conventions.md](references/project-conventions.md) before designing or reviewing a feature.

## Workflow

1. Inspect `values.yaml`, related templates/helpers, representative examples, rendered manifests, and `git status` before deciding the contract.
2. Identify the resource owner and place optional configuration next to it. Prefer workload-scoped blocks when the resource exists to expose a workload.
3. Design the common path as expressive DDL. Use scalar or compact map forms when they remove repetitive Kubernetes structure.
4. Infer stable relationships such as local workload name, namespace, labels, Service name, and named Service ports. Require explicit values when inference could silently choose the wrong target.
5. Provide `extras` at the smallest useful native API boundary. Define which generated fields override extras and which extras intentionally replace generated structures.
6. Render nothing when the optional block is absent. Validate invalid operation/resource combinations and ambiguous or missing references with clear `required` errors.
7. Implement using existing scope, fullname, labels, merge, and iteration patterns. Avoid unrelated refactors and preserve user changes.
8. Add a focused values example and its `helm template` output under `examples/`.
9. Validate positive, absent-resource, override, and negative cases. Render all existing input examples and report preexisting failures separately.

## DDL Decisions

- Optimize for the common local-workload case; permit explicit external targets as an override.
- Prefer named references over copied numeric values when another DDL field already owns the value. Resolve and validate the reference during rendering.
- Keep Kubernetes-native advanced configuration under `extras`, rather than duplicating the full upstream API.
- Preserve upstream defaults by omitting default-valued fields unless emitting them improves the DDL contract or supports a custom target.
- Treat cross-namespace references as explicit configuration and document external prerequisites such as `ReferenceGrant`.
- Keep one resource per workload by default unless multiple resources solve a demonstrated requirement.

## Template Rules

- Derive operation with `default $key .operation` and gate resource generation with `.enabled` plus the applicable operation.
- Build workload scope with `mustMergeOverwrite (deepCopy .) $ (dict "scope" $key)` before calling shared helpers.
- Use `deepCopy` before merges or `set` operations so template processing does not mutate values reused elsewhere.
- Reuse `kubecraft.app-fullname` and `kubecraft.metadata-labels` for generated resources.
- Let chart-owned identity and selector fields win over `extras`. Permit documented native collection overrides, such as an explicit `backendRefs` replacing an inferred backend.
- Write validation messages with the feature name, workload key, and rule/item index when applicable.

## Validation

Run at least:

```bash
helm lint . -f examples/<feature>.yaml
helm template <release> . -f examples/<feature>.yaml
helm template <release> . -f examples/<feature>.yaml --set <optional-block>=null
```

Also exercise malformed operation, missing required reference, unknown named reference, and native override cases. Confirm the stored rendered manifest matches a fresh render. Render every non-`*-manifests.yaml` example and distinguish failures introduced by the change from failures already present in the worktree.
